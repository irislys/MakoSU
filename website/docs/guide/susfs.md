# SuSFS 使用

MakoSU Manager 提供 SuSFS 用户空间配置界面，但可用能力由设备内核决定。Manager 不能为没有集成 SuSFS 的内核增加内核侧功能。

## 使用前提

- Manager 首页已经正确识别 MakoSU 内核与 Root 状态。
- SuSFS 页面能够读取“已启用功能”，且不是空白或全不支持。
- 当前内核明确集成了与 Manager 兼容的 SuSFS 版本。
- 修改前已经创建配置备份，并保留能正常启动的镜像。

## 推荐配置顺序

1. 进入 SuSFS 页面，先查看内核报告的已启用功能。
2. 创建一次完整备份，记录当前可用状态。
3. 每次只修改一类配置，例如隐藏路径或 Maps 规则。
4. 应用配置后检查结果，再决定是否启用自动启动。
5. 需要重启的项目完成重启后，验证目标应用和系统稳定性。
6. 出现异常时先恢复备份，不要继续叠加更多规则。

## 配置分类

### 隐藏路径

用于从目标进程视角隐藏指定路径。路径必须真实、明确，并避免隐藏系统启动和目标应用正常运行所需的目录。过宽规则会造成应用异常或系统功能缺失。

### Maps 与内存信息

用于调整目标进程可见的内存映射信息。规则应针对已确认的检测项，不要批量添加来源不明的模板。

### Kstat

用于为指定路径配置文件状态信息。伪造值必须内部一致；互相矛盾的时间、大小或 inode 信息可能比原始信息更容易触发异常。

### uname 与构建时间

修改应用可见的内核版本或构建时间。该功能不会改变真实内核，也不能修复 KMI 不匹配。用于兼容性判断时始终以真实内核信息为准。

### 自动启动

自动启动会在启动阶段应用非默认配置。MakoSU 使用暂存、同步、切换和失败回滚流程降低损坏风险，但错误规则仍可能让应用或系统服务异常。确认手动应用稳定后再启用。

## 保存与恢复

MakoSU 的当前用户空间实现会：

- 使用跨进程锁防止并发更新相互覆盖。
- 通过临时文件、`fsync` 和原子替换提交配置。
- 拒绝截断数据、重复键、超限字段与尾随数据。
- 通过一次 Root 调用读取完整配置，减少界面等待。
- 在自动启动模块更新失败时尝试恢复旧配置与旧模块状态。

备份恢复会覆盖当前设置。导入前核对备份来源、设备与版本；版本不一致时先保留当前备份，再决定是否尝试恢复。

## 修改后没有生效

依次检查：

1. 内核是否报告对应 SuSFS 功能已启用。
2. 规则是否已保存，输入是否包含当前格式不支持的分隔符。
3. 该设置是否要求重启设备。
4. 自动启动是否启用，启动阶段是否出现回滚。
5. 目标应用是否使用了独立进程、隔离服务或不同用户空间。

仍无法确认时，先恢复最小配置，再按[故障排查](/guide/troubleshooting)采集日志。
